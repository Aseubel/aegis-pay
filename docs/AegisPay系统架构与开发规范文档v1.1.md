# AegisPay (神盾支付) 系统架构与开发规范文档

## 文档概述

本文档为 **AegisPay** 支付网关系统的标准架构设计与开发规范。本项目采用 **Go 语言单体架构 (Monolithic)**，严格遵循 **领域驱动设计 (DDD - Domain Driven Design)** 思想。

系统旨在提供一个高可用、防资损、易扩展的统一支付中台，并深度融合 LangChain-Go 赋能商户数据治理与智能风控。

**核心准则：领域层（Domain）是系统的绝对核心，必须保持纯洁，严禁泄漏任何与基础设施相关的实现细节。在追求轻量化的同时，必须为千万级并发与极端异常场景预留防御性设计。**

## 一、 系统架构与 DDD 分层模型

本项目虽然是单体应用，但内部严格划分为四层架构（依赖方向严格由外向内）：

1. **User Interface 层 (用户接口层 /** **`interfaces`)**
   - 负责接收外部请求和数据校验（HTTP API、Webhook、WebSocket）。
2. **Application 层 (应用服务层 /** **`application`)**
   - 负责用例流转，是领域服务的编排者。处理事务控制、分布式锁获取、领域事件发布。
3. **Domain 层 (领域层 /** **`domain`) - 核心！**
   - 包含业务实体 (Entity)、值对象 (Value Object)、领域服务与防腐层接口 (ACL)。
   - **支付 ACL 准则**：支付业务异构性极强（如微信混杂 XML/JSON，支付宝纯 JSON）。防腐层的核心使命是**统一领域模型转换**，将外部千奇百怪的报文严格映射为内部纯净的 `PaymentOrder`，这是支付系统必须承受且极具价值的“好成本”。
4. **Infrastructure 层 (基础设施层 /** **`infrastructure`)**
   - 负责实现 Domain 层定义的接口（MySQL, Redis, Milvus, LangChain-Go, 支付渠道 SDK）。

## 二、 核心领域划分 (Bounded Contexts)

1. **Transaction 领域 (交易核心域)**：维护支付状态机，生成全局唯一流水号。
2. **Channel 领域 (渠道路由域)**：屏蔽底层第三方支付差异，提供动态路由与降级能力。
3. **Merchant 领域 (商户支撑域)**：商户进件、RSA/MD5 验签、异步通知。
4. **Copilot 领域 (AI 智能与风控域)**：
   - **NL2SQL**：商户自然语言报表查询。
   - **RAG 智能风控**：异步旁路风控诊断。

## 三、 金融级防资损与高并发控制方案

### 1. 双重幂等防御体系 (防重复入账)

- **外层防御（Redis 分布式锁）**：以 `TradeNo` 为 Key 拦截并发到达的回调请求，过滤 99% 的无效计算。
- **底线兜底（状态机乐观锁）**：`UPDATE status='SUCCESS' WHERE trade_no=? AND status='PROCESSING'`。无论 Redis 锁是否因宕机失效，底层 DB 绝对保证状态单向不可逆，防击穿。

### 2. 高可靠异步队列演进 (Redis Streams -> RocketMQ)

- **当前架构 (Redis Streams)**：采用 Redis 5.0+ Streams + 消费组 (PEL) 实现轻量级 Webhook 阶梯重试。为防 OOM，**必须强制配置** **`MAXLEN ~ 100000`**，且强烈建议与核心缓存 Redis **实例物理隔离**。
- **演进路线**：当系统吞吐量跨越千万级，商户通知系统应平滑替换为 RocketMQ，利用其原生的事务消息与海量堆积能力。

### 3. 分布式环境下的“掉单”自愈机制

- **拉模式主动补偿**：后台 Cron Job 定期扫描超 N 分钟仍为 `PROCESSING` 的订单，主动向第三方网关发起 `QueryOrder` 查单补偿。
- **T+1 终态对账**：每日自动拉取微信/支付宝前一日交易账单，与本地流水进行全量双向对账，精准捕获长尾掉单并生成差异报表。

## 四、 大模型 (LLM) 安全与时效性架构设计

在金融系统中引入 AI，必须解决“慢查询瘫痪系统”、“越权越界”、“时效性矛盾”三大致命问题。

### 1. NL2SQL 的三维安全隔离防火墙 (防 DoS 与越权)

商户通过自然语言生成 SQL 极易产生联合查询（UNION）越权或笛卡尔积慢查询，必须采用硬隔离：

- **数据引擎隔离**：AI 生成的 SQL 仅允许路由至**只读从库 (Read-Replica)**。
- **数据库级 RLS (行级安全策略)**：放弃脆弱的 AST 代码级校验。为每次 AI 查询动态生成**绑定当前租户 (Tenant\_ID) 权限的临时 DB User**，或利用 PostgreSQL/MySQL 的 RLS 特性，从数据库引擎底层彻底杜绝跨商户越权（即便 LLM 生成了 UNION 也无法查出越权数据）。
- **资源配额与熔断 (Kill Query)**：为 AI 服务分配**完全独立的极小连接池**，并强制附加 `context.WithTimeout`（如 3 秒）。一旦超时，在 DB 层发起 `KILL <processlist_id>`，防止慢查询拖垮从库 CPU。

### 2. RAG 异步风控的“先放后杀”策略 (解决时效性矛盾)

支付主链路要求 <50ms 响应，而 RAG 大模型推理耗时在秒级/分钟级。

- **混合风控架构**：主链路采用极速 C++ 规则引擎（拦截明显黑产）。RAG 大模型则作为**旁路异步检测**，负责从历史向量库中挖掘“新型欺诈特征”。
- **T+1 资金拦截逻辑**：由于国内支付普遍采用 T+1 结算（次日到账），大模型即便晚几分钟得出高危诊断报告，系统仍有充足时间发起\*\*“商户账户冻结”\*\*。这种“先放行交易，后拦截提现（先放后杀）”的策略，完美化解了 AI 延迟与金融时效性之间的矛盾。

## 五、 代码开发 SOP (AI 生成指令规范)

任何新增模块的代码生成，必须严格遵循 DDD 纯洁性：

1. **Domain 优先**：先定义 Entity（如 `RiskReport`）与 Gateway 接口，全程禁止 Import `gorm` 或 `langchaingo`。
2. **Infrastructure 承接**：在基础设施层实现接口。大模型相关的 API 请求、Prompt 组装、向量检索全部封装于 `ai_adapter` 目录下。
3. **Application 编排**：应用层通过注入的接口进行调度。对于耗时的 LLM 任务，必须在 Application 层封装为异步 Goroutine 或投递至 Redis Streams，严禁阻塞 HTTP 请求线程。
